## 一、开发总则

### 1.1 先 API 后框架

依据 Anthropic "Building Effective Agents" 的核心建议：**所有组件的第一版必须直接用 LLM API 实现，不引入任何 Agent 框架。**

```
开发流程：
  LLM API 直接调用 → 验证可行性 → 评估通过 → 考虑框架加速
                                         ↓ 不通过
                                    重新设计 Prompt
```

**具体执行规则：**
1. 6 个组件（共享记忆、工作流编排、评估闭环、人机审批、安全可观测、经验学习）的 MVP 均使用 Python + HTTP 调用 LLM API
2. 只有在该组件通过 Phase 1 评估 Gate 后，才评估是否引入框架
3. 引入框架前需回答三个问题：收益是否大于抽象泄漏成本？调试难度是否可控？框架锁定风险是否可接受？

**禁止行为：**
- 禁止在 MVP 阶段引入 LangChain/LlamaIndex/CrewAI 等框架
- 禁止因为"框架有现成的"而跳过直接 API 实现的验证
- 禁止在框架之上再套一层自研抽象（除非有明确的多个框架切换需求）

### 1.2 开发环境与工具链

| 项 | 选型 | 理由 |
|----|------|------|
| 语言 | Python 3.12+ | 与现有 OpenClaw 生态一致 |
| 包管理 | pip + venv | 简单够用，不需要 poetry 的复杂度 |
| 类型检查 | mypy (strict mode) | 所有公共接口必须有类型注解 |
| 格式化 | ruff format + ruff check | 替代 black + isort + flake8 |
| 测试 | pytest + pytest-asyncio | 异步 Agent 代码需要 async 测试支持 |
| 日志 | structlog | 结构化日志，JSON 格式输出到文件 |
| 配置 | python-dotenv + pydantic-settings | 环境变量 → 结构化配置对象 |

## 二、组件 1：共享记忆系统 — 开发规范

### 2.1 核心接口：MemoryGateway

```python
# src/memory/gateway.py

from typing import Protocol, Optional
from src.shared.models import MemoryEntry, MemoryQuery, MemorySearchResult

class MemoryGateway(Protocol):
    """记忆系统统一入口。所有 Agent 通过此接口读写记忆。"""

    async def write(self, entry: MemoryEntry, *, trace_id: str) -> str:
        """写入记忆条目，返回 entry_id。内部走 Write Policy 评估。"""
        ...

    async def read(self, entry_id: str, *, agent_id: str, trace_id: str) -> Optional[MemoryEntry]:
        """按 ID 读取单条记忆。自动更新 access_count 和 last_accessed_at。"""
        ...

    async def search(self, query: MemoryQuery, *, trace_id: str) -> list[MemorySearchResult]:
        """语义搜索 + 关键词搜索 → 合并去重 → 重排序 → 返回 top_k。"""
        ...

    async def consolidate(self, *, agent_id: str, session_id: str, trace_id: str) -> list[str]:
        """将工作记忆中 importance >= 4 的条目提升到长期记忆。"""
        ...

    async def decay(self, *, layer: str, trace_id: str) -> int:
        """运行遗忘策略，返回本次淘汰的条目数。"""
        ...
```

### 2.2 存储后端

| 层级 | 后端 | Schema |
|------|------|--------|
| 短期记忆 | LLM 上下文窗口 | 系统消息注入块 |
| 工作记忆 | `dict[str, MemoryEntry]`（内存）+ SQLite 备份 | key = entry_id |
| 长期记忆 | ChromaDB（向量）+ SQLite（元数据） | 见下方 DDL |

```sql
-- 长期记忆元数据表
CREATE TABLE memories (
    id TEXT PRIMARY KEY,                -- UUID v4
    agent_id TEXT NOT NULL,             -- 写入 Agent ID
    content TEXT NOT NULL,              -- 记忆内容（≤2000 字符）
    memory_type TEXT NOT NULL,          -- fact/decision/experience/preference/task_context
    importance REAL NOT NULL DEFAULT 0.5, -- 0-1
    tags TEXT,                          -- JSON array
    source_trace_id TEXT,               -- 来源 trace
    created_at TEXT NOT NULL,           -- ISO 8601
    last_accessed_at TEXT NOT NULL,
    access_count INTEGER DEFAULT 0,
    expires_at TEXT,                    -- NULL = 永不过期
    chroma_id TEXT,                     -- ChromaDB 中对应的 document id
    UNIQUE(chroma_id)
);

CREATE INDEX idx_memories_agent ON memories(agent_id);
CREATE INDEX idx_memories_type ON memories(memory_type);
CREATE INDEX idx_memories_importance ON memories(importance DESC);
CREATE INDEX idx_memories_expires ON memories(expires_at);
```

### 2.3 关键实现细节

**查询流程（多路检索 + 重排序）：**

```
                     Query
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
    ChromaDB      SQLite FTS    Tag Match
    语义检索      关键词检索      标签过滤
    top_k×3       top_k×3       top_k×3
          │            │            │
          └────────────┼────────────┘
                       ▼
                  合并去重
                 (by content hash)
                       │
                  候选 > top_k？
                  ├─ No → 直接返回
                  └─ Yes → LLM 重排序 → 返回 top_k
```

**遗忘策略公式：**

```
retain_score = importance × 0.6 + freshness × 0.2 + access_freq × 0.2

freshness = 1 / (1 + days_since_last_access / 30)
access_freq = min(access_count / 10, 1.0)

IF importance > 0.9 OR access_count > 10 → 永远保留
ELSE IF retain_score < 0.3 → 标记删除（归档到 memory_archive 表后删除）
```

## 三、组件 2：工作流编排引擎 — 开发规范

### 3.1 YAML 工作流定义规范

工作流定义文件存放在 `config/workflows/`，命名规则 `{name}-v{version}.yaml`。

```yaml
# 最小工作流模板
workflow:
  name: "my-workflow"
  version: "1.0"
  description: "一句话描述"
  trigger:
    type: manual              # manual | schedule | event
    config: {}                # schedule: {cron: "0 8 * * *"}, event: {event_name: "..."}

  params:                     # 工作流参数（外部传入）
    topic:
      type: string
      required: true
      default: "AI Agent"

  steps:
    - id: step-1
      name: "步骤名称"
      agent: researcher       # 执行 Agent ID
      type: task              # task | condition | parallel | loop
      depends_on: []          # 依赖的步骤 ID 列表
      input:                  # 输入模板，支持 $变量引用
        query: "$params.topic"
      config:
        prompt: "搜索 {{query}} 的最新信息"
        tools: [web_search, web_fetch]
        timeout: 300          # 秒
        retry:
          max_attempts: 2
          backoff: exponential
      output_as: search_result  # 步骤输出在 context 中的 key

    - id: step-2
      name: "条件路由"
      type: condition
      depends_on: [step-1]
      condition:
        expression: "$search_result.count > 0"
        then: step-3
        else: step-fallback

    - id: step-3
      name: "并行处理"
      type: parallel
      depends_on: [step-2]
      parallel:
        branches: [step-3a, step-3b]
        merge: all            # all | first | vote
```

### 3.2 引擎核心接口

```python
# src/workflow/engine.py

class WorkflowEngine:
    """工作流编排引擎。不依赖任何 Agent 框架，直接用 asyncio 实现。"""

    async def load(self, workflow_path: str) -> Workflow:
        """从 YAML 加载工作流定义，校验 schema。"""

    async def run(self, workflow: Workflow, params: dict, *, trace_id: str) -> WorkflowResult:
        """
        执行工作流。
        1. 解析 params → 注入到 context
        2. 按拓扑排序执行 steps
        3. 每个 step 执行前后触发 PreStep/PostStep hooks
        4. 记录每条 step 的输入/输出/耗时到 trace
        5. 异常时按步骤的 retry 配置重试，全部失败则标记 partial_success
        """

    async def resume(self, run_id: str, *, from_step: str, trace_id: str) -> WorkflowResult:
        """从指定步骤恢复执行（用于人工审批后继续）。"""
```

### 3.3 步骤执行器

```python
# src/workflow/steps.py

class StepExecutor:
    """步骤执行器，处理四种步骤类型。"""

    async def execute(self, step: Step, context: dict, *, trace_id: str) -> StepResult:
        match step.type:
            case "task":
                return await self._execute_task(step, context, trace_id)
            case "condition":
                return await self._execute_condition(step, context, trace_id)
            case "parallel":
                return await self._execute_parallel(step, context, trace_id)
            case "loop":
                return await self._execute_loop(step, context, trace_id)

    async def _execute_task(self, step, context, trace_id):
        """向目标 Agent 发送任务，等待结果。超时按 retry 配置处理。"""
        ...

    async def _execute_condition(self, step, context, trace_id):
        """解析条件表达式，返回 then/else 分支的下一步步骤 ID。"""
        ...
```

## 五、组件 4：人机协作审批 — 开发规范

### 5.1 风险等级定义

```python
# src/approval/router.py

class RiskLevel(Enum):
    L0 = 0    # 读操作：搜索、读文件、查记忆 → 自动通过，事后日志
    L1 = 1    # 安全写：写文件、发消息、写记忆 → 自动通过，事后通知
    L2 = 2    # 风险写：修改配置、调付费 API、发全员消息 → 飞书确认卡片
    L3 = 3    # 危险操作：删文件、改数据库、rm -rf、DROP → 飞书确认 + 确认码

# 操作 → 风险等级 映射表
OPERATION_RISK_MAP = {
    "read": RiskLevel.L0,
    "write_file": RiskLevel.L1,
    "memory_write": RiskLevel.L1,
    "send_message": RiskLevel.L1,
    "modify_config": RiskLevel.L2,
    "call_paid_api": RiskLevel.L2,
    "broadcast_message": RiskLevel.L2,
    "delete_file": RiskLevel.L3,
    "drop_table": RiskLevel.L3,
    "rm_rf": RiskLevel.L3,
    "modify_system_prompt": RiskLevel.L3,  # 红线：绝对禁止
}
```

### 5.2 审批流实现

```
L2/L3 操作触发：
  1. PreToolUse hook 拦截 → 查询 OPERATION_RISK_MAP
  2. 风险 L2/L3 → 暂停 Agent 执行
  3. 生成飞书审批卡片 → 发送给阿禹
  4. 等待审批（超时 30min → 自动拒绝）
  5. 审批通过 → 继续执行；拒绝 → 返回拒绝原因给 Agent
```

### 6.2 Prompt 注入防御

```python
# src/safety/sanitize.py

# 必须拦截的注入模式
BLOCKED_PATTERNS = [
    r"忽略.*(?:指令|规则|系统提示|system prompt|以上)",
    r"ignore.*(?:instructions|rules|system|above)",
```

## 七、组件 6：经验学习闭环 — 开发规范

### 7.1 经验提取流程

```
任务完成/失败
  → ExperienceExtractor.extract(task_log, trajectory)
    → LLM 分析：失败原因 / 成功策略 / 关键决策
    → 生成 Experience 条目
  → ExperienceValidator.validate(experience)
    → 安全检查（无 PII、无 System Prompt 片段）
    → 去重检查（相似经验合并）
    → 推送给阿禹确认
  → 确认后写入记忆库（memory_type: "experience"）
  → 下次同类任务启动时自动注入
```

### 7.2 经验 Schema

```python
class Experience(BaseModel):
    id: str
    agent_id: str
    task_type: str              # transcription | code_generation | data_collection | ...
    trigger: str                # 触发条件："音频时长 > 5 分钟"
    symptom: str                # 现象："Whisper 转写超时"
    root_cause: str             # 根因："M1 Air CPU 处理 30 分钟音频时内存不足"
    solution: str               # 解决方案："ffmpeg 拆分为 60s 片段，逐段转写后拼接"
    outcome: str                # 结果："成功转写 30 分钟视频"
    confidence: float           # 0-1，基于验证次数
    validated: bool             # 是否经人工确认
    created_at: datetime
    last_applied_at: datetime | None
    apply_count: int            # 被引用的次数
```

## 九、沙箱环境配置

### 9.1 分层隔离

| 层级 | 隔离方式 | 适用场景 |
|------|---------|---------|
| Level 1 | Python subprocess + timeout | 一般工具调用（搜索、读文件） |
| Level 2 | Docker container + tmpfs | 代码执行、脚本运行 |
| Level 3 | 独立 VM / macOS sandbox | 自主 Agent 完整运行 |

Phase 1 使用 Level 1，Phase 2 升级到 Level 2，Phase 3 评估是否需要 Level 3。

### 9.2 工具调用沙箱

```python
# 所有 Agent 的工具调用必须经过沙箱
async def sandboxed_tool_call(
    agent_id: str,
    tool_name: str,
    params: dict,
    *,
    timeout: int = 30,
    trace_id: str,
) -> Any:
    """
    在隔离环境中执行工具调用。
    - subprocess 执行，超时强制 kill
    - 限制文件系统访问范围（只能读写 workspace/ 目录）
    - 限制网络访问（只能访问白名单域名）
    - 限制资源使用（CPU time < 60s, memory < 1GB）
    """
```

## 十、MCP 集成指南

### 10.1 工具暴露方式

所有新增工具优先通过 MCP (Model Context Protocol) 暴露给 Agent。现有 CC-Connect 工具保持不变。

```python
    """将重要信息写入共享记忆库。"""
    ...
```

### 10.2 MCP 优先原则

- 新工具必须通过 MCP 暴露（提供标准化的 tool description schema）
- 现有工具逐步迁移到 MCP（Phase 2 完成）
- Agent 不直接调用 Python 函数，必须通过 MCP tool 接口
