## 一、架构模式选择

### 总体模式：Orchestrator-Workers + Evaluator-Optimizer 混合

```
┌─────────────────────────────────────────────────────────────┐
│                    编排器 (Orchestrator)                      │
│                唯一的 Autonomous Agent                       │
│              读取工作流 YAML → 按序调度 Worker               │
│              监控每步输出 → 异常路由到审批                    │
└──────┬──────────────────────────────────────────────────────┘
       │
       ├──── Worker: researcher (采集→提取→入库)
       ├──── Worker: tech-dev (编码→测试→部署)
       ├──── Worker: copywriter (文案生成)
       ├──── Worker: script-editor (脚本编辑)
       ├──── Worker: data-analyst (数据分析)
       ├──── Worker: visual-designer (视觉生成)
       ├──── Worker: product-designer (产品设计)
       ├──── Worker: ops-monitor (监控→告警)
       ├──── Worker: investment-analyst (投资分析)
       └──── Worker: main/陈陈 (汇总→汇报→审批路由)
```

**选型理由：**

| 模式 | 选否 | 理由 |
|------|------|------|
| Prompt Chaining | 部分采用 | 单个 Agent 内部使用（采集→提取→入库 就是链式调用），但全局编排不够 |
| Routing | 采用（审批系统） | 操作风险等级路由到不同的审批路径 |
| Parallelization | 部分采用 | 多个独立 Agent 可同时执行（如 researcher 和 copywriter 并行） |
| Orchestrator-Workers | **主模式** | 11 个 Agent 是 Worker，需要中央编排器管理任务分配和状态 |
| Evaluator-Optimizer | 采用（评估闭环） | 任务完成后 LLM 评估→结果写入记忆→下次优化 |

> 依据 Anthropic: Orchestrator-Workers 适用于"can't predict subtasks ahead of time"的场景。本项目中阿禹的任务多种多样，无法预定义所有子任务，编排器需要动态分解。

## 二、渐进式复杂度

**原则：从简单到复杂，每一步都有退出标准，验证不通过就不升级。**

### 演进路径

```
Level 0: 单 Agent 无记忆
  └─ 退出标准：单个 Agent 独立完成任务，通过 Happy Path 测试

Level 1: + 共享记忆注入
  └─ Agent 启动时注入 3 条相关历史经验
  └─ 退出标准：同类任务执行 3 次后，第 4 次 agent 能引用历史经验

Level 2: + 经验学习闭环
  └─ 任务完成/失败后自动提取经验 → 人工确认 → 写入记忆
  └─ 退出标准：一周内自动积累了 10+ 条有效经验，人工确认率 > 80%

Level 3: + 安全护栏
  └─ 所有工具调用经过 PreToolUse hook 拦截
  └─ L2/L3 操作触发飞书审批
  └─ 退出标准：Adversarial 测试集全部拦截成功，零越权

Level 4: + 工作流编排
  └─ YAML 定义流水线，编排器自动调度
  └─ 退出标准：2 条流水线持续运行 24h 不间断

Level 5: + 评估闭环
  └─ 每次任务后自动评分，阿禹抽查
  └─ 退出标准：自动评估与人工评估一致率 > 80%

Level 6: + 数据飞轮
  └─ 生产失败自动回流到标注队列
  └─ 退出标准：每月从失败中自动生成了 5+ 条回归测试用例
```

**关键决策点：**
- 如果 Level 1 的记忆注入已经足够好用，就不需要 Level 6 的数据飞轮
- 如果 Level 3 的安全护栏导致了审批疲劳（每天 > 20 次审批），需要回到 Level 2 调整风险分级

## 三、工具设计规范

每个工具必须遵守以下规范（依据 Anthropic ACI 设计原则）：

### 工具定义模板

```yaml
tool:
  name: memory_search          # 唯一标识，snake_case
  description: |
    在共享记忆库中搜索与当前任务相关的历史经验。
    返回按相关性排序的记忆条目列表。
    适用场景：Agent 启动时检索相关经验，任务执行前查历史。
    不适用场景：实时数据查询、外部信息搜索（请用 WebSearch）。
  parameters:
    query:
      type: string
      required: true
      description: 搜索查询词。建议使用关键词组合，不要用完整句子。
      example: "whisper transcription timeout audio chunking"
    agent_id:
      type: string
      required: true
      description: 当前 Agent 的 ID。用于过滤与当前 Agent 角色相关的记忆。
    top_k:
      type: integer
      required: false
      default: 5
      range: [1, 20]
      description: 返回前 K 条结果
  returns:
    type: array
    items:
      type: object
      properties:
        text: {type: string, description: 记忆文本内容}
        score: {type: float, description: 语义相似度 [0, 1]}
        source: {type: string, description: 来源标识}
        created_at: {type: string, description: ISO8601 时间戳}
  errors:
    - code: MEMORY_STORE_UNAVAILABLE
      description: ChromaDB 连接失败
      action: retry 3 次指数退避，失败后降级为不注入记忆继续工作
    - code: EMPTY_RESULT
      description: 无相关记忆
      action: 正常返回空数组，不触发重试
```

### 工具开发原则

1. **描述比实现更重要** — 工具 description 是 LLM 选择工具的唯一线索，歧义描述 → 误用
2. **参数必须有 example** — LLM 对参数格式的理解来源 example，不是 type annotation
3. **错误状态必须明确** — 每个可能的错误码都要告诉 Agent 怎么处理（重试/跳过/升级）
4. **工具数量 < 20** — 超过 20 个工具 LLM 选择准确率显著下降

## 四、记忆架构

### 三层记忆模型

```
┌─────────────────────────────────────────────────────────────┐
│                    短期记忆（上下文窗口）                       │
│  ├─ 会话历史（当前对话）                                       │
│  ├─ 注入的相关经验（启动时检索 top 5）                         │
│  ├─ 注入的 Agent 角色定义                                     │
│  └─ 生命周期：单次会话                                         │
├─────────────────────────────────────────────────────────────┤
│                    工作记忆（临时状态）                         │
│  ├─ 当前任务状态（进行中/等待审批/已完成）                      │
│  ├─ 工具调用计数器（防死循环，> 3 次同工具→告警）               │
│  ├─ Relay 消息队列                                            │
│  └─ 生命周期：单次任务 → 任务结束清理                           │
├─────────────────────────────────────────────────────────────┤
│                    长期记忆（持久化）                           │
│  ├─ ChromaDB: 语义记忆（经验条目、成功模式、教训）             │
│  ├─ SQLite: 结构化记忆（任务日志、评估记录、审批历史）          │
│  ├─ knowledge/: 静态知识（分类体系、概念文档）                  │
│  └─ 生命周期：持久，带智能淘汰                                  │
└─────────────────────────────────────────────────────────────┘
```

### 记忆生命周期

```
写入：任务完成后 30s 内提取经验 → 写入 ChromaDB
检索：Agent 启动时检索 top 5 + 任务开始时按 task_type 检索
更新：同类型经验合并去重（相似度 > 0.85 时合并，更新 timestamp）
遗忘：
  - 60 天未访问 → 标记 stale
  - 90 天未访问 + 非标星 → 自动删除
  - 标星条目 → 永不过期
验证：人工确认过的记忆优先级 > 未确认的
```

### 记忆安全性

- Agent 只能写"自己的经验"（agent_id 过滤）
- Agent 不能删除记忆（delete 仅开放给阿禹 + main Agent）
- 记忆写入前经内容安全过滤（去 PII、去系统 Prompt 片段）
- 记忆检索结果带 source 标注（追溯来源）

## 五、规划策略

### 默认策略：ReAct（Reasoning + Acting）

```
Task → Thought → Action → Observation → Thought → Action → ... → Answer
```

所有 Agent 默认使用 ReAct 循环。单个 Agent 内部的推理链路：
1. Thought: 分析当前任务需要什么
2. Action: 调用工具（搜索、查记忆、写文件）
3. Observation: 解读工具返回结果
4. 循环直到任务完成或超时

### 复杂决策场景：Reflexion

当 ReAct 循环 > 5 步仍无结果时，自动切换到 Reflexion 模式：
- 回顾前面步骤 → 发现策略错误 → 调整方向 → 重试

### 规划约束（防 problems 文档中的典型失败）

| 约束 | 防止的问题 |
|------|-----------|
| 推理链 ≤ 7 步 | 1.1.1 #2 "多步推理退化" |
| 同工具调用 ≤ 3 次 | 1.3.1 #4 "重复调用" |
| 单次任务 token 消耗 ≤ 50K | 6.1 #1 "推理成本失控" |
| 15 分钟超时自动终止 | 6.2 #3 "长推理链超时" |
| 不确定性时输出置信度 | 1.1.1 #13 "不确定性量化困难" |

## 六、错误恢复设计

### 分级错误处理

```
L1 可重试错误（临时性）
  ├─ API 超时 / 限流 (429)
  ├─ 网络错误
  ├─ 工具返回格式异常（单次）
  └─ 策略：指数退避重试 3 次，间隔 [2s, 4s, 8s]
      3 次均失败 → 升级到 L2

L2 需调整策略的错误（可恢复）
  ├─ 工具调用参数错误
  ├─ 文件不存在 → 检查路径
  ├─ 记忆检索无结果 → 换关键词
  └─ 策略：ReAct 反思 → 修改策略 → 重试 1 次
      仍失败 → 升级到 L3

L3 需人工介入的错误（不可自动恢复）
  ├─ 权限不足
  ├─ 数据不一致
  ├─ 逻辑矛盾
  ├─ 同工具连续 3 次失败
  └─ 策略：生成错误摘要 → 发送给 main Agent → 通知阿禹
      等待人工指示
```

### 多 Agent 协作错误处理

| 场景 | 处理 |
|------|------|
| Worker Agent 超时未响应 | 编排器等待 120s → 超时 → 跳过该 step → 标注为 partial_success |
| Relay 消息丢失 | 发送方等 30s 无 ack → 重发 → 两次无 ack → 记录到日志 + 通知 ops-monitor |
| 协作死锁（A 等 B、B 等 A） | 每个 relay 调用带 `depends_on` 字段 → 编排器检测循环依赖 → 阻断 |
| 级联失败 | 上游 Agent 失败后，下游 Agent 收到 `input_status: failed` → 暂停执行 → 通知阿禹 |

## 七、安全架构

### 分层防御模型

```
Layer 1: 输入安全
  ├─ 所有用户输入经过 sanitize（去除 Prompt 注入特征）
  ├─ RAG 检索结果中的指令与系统 Prompt 隔离（不同 context 分区）
  └─ 工具返回结果过滤（识别并标记可疑内容）

Layer 2: 行为安全
  ├─ PreToolUse hook: 每次工具调用前检查权限
  ├─ PostToolUse hook: 每次工具调用后检查结果是否可疑
  ├─ Stop hook: 任务终止前写入审计日志
  └─ 每个 hook 可设置: allow / block / escalate

Layer 3: 审批安全
  ├─ L0: 读操作 → 自动通过
  ├─ L1: 安全写 → 自动通过 + 日志
  ├─ L2: 风险写 → 需飞书确认
  └─ L3: 危险操作 → 需确认码

Layer 4: 审计安全
  ├─ 每条操作记录 JSONL 日志
  ├─ 日志格式: {ts, agent_id, tool, params_hash, result_code, tokens, duration_ms}
  └─ 日志不可被 Agent 修改（写入权限只有 ops-monitor 拥有）
```

### Prompt 注入防御

```python
# 输入 sanitize 规则
BLOCKED_PATTERNS = [
    r"忽略.*(?:指令|规则|系统提示|system prompt)",
    r"ignore.*(?:instructions|rules|system)",
    r"你是.*(?:开发者|DAN|开发者模式)",
    r"you are.*(?:developer|DAN|dev mode)",
    r"\[SYSTEM\]:.*",
    r"<\|im_start\|>system",
]

# 系统 Prompt 与用户输入隔离
# 系统 Prompt 在最外层，用户输入在 <user_input> 标签内
# RAG 检索结果在 <retrieved_context> 标签内
# 工具返回在 <tool_result> 标签内
# 每个区域的指令不能跨越到其他区域
```

## 八、模型选型

### 模型策略

```
主力模型: MIMO v2.5 Pro（推理 + 长上下文 1M）
图片模型: MIMO v2.5（支持 text+image 输入）
降级模型: DeepSeek V4 Pro
语音模型: MIMO v2.5 TTS（备用）

路由策略（Phase 3 实现）：
  简单操作（搜索/检索/状态查询） → 小模型（mimo-v2.5）
  复杂推理（编码/分析/评估） → 大模型（mimo-v2.5-pro）
  图片任务 → MIMO v2.5（多模态）
  紧急降级 → DeepSeek V4 Pro

成本控制：
  每 Agent 每日 token 预算: 50K
  超出后自动降级为仅响应关键操作 + 通知阿禹
```
