## 一、Agent 选型决策

**结论：确实需要 Agent 系统，但不是每个组件都需要 Agent 自主性。**

| 组件 | 选型 | 理由 |
|------|------|------|
| 共享记忆系统 | 工作流（Workflow） | 预定义路径：写入→索引→检索→注入。不需要 LLM 动态决策 |
| 工作流编排引擎 | 工作流 | 预定义 YAML 驱动，状态机执行。不需要 Agent 自主改工作流 |
| 评估反馈闭环 | 评估器-优化器（Evaluator-Optimizer） | LLM 做评估，结果写入记忆，但优化由人工确认 |
| 人机协作审批 | 路由（Routing） | 根据操作风险等级路由到人工审批或自动通过 |
| 安全可观测性 | 工作流 | 纯日志+监控，零 AI 参与。确定性逻辑 |
| 经验学习闭环 | 评估器-优化器 | LLM 提取经验，人工确认后写入，Agent 启动时自动注入 |

> 依据 Anthropic "Building Effective AI Agents": "For many applications, optimizing single LLM calls with retrieval and in-context examples is usually enough." 本项目中记忆注入、经验检索、审批路由都是单次 LLM 调用级别的操作，不需要全自主 Agent 循环。唯一需要 Agent 模式的场景是：多 Agent 之间 relay 通信后自动决策下一步（编排器是唯一的 Agent）。

## 二、自主度定义

每个 Agent 的自主等级，依据 Anthropic 三级模型：

| Agent | 自主等级 | 说明 |
|-------|---------|------|
| main/陈陈 | State Machine | 固定工作流但可以路由决策（选择派哪个 Agent） |
| researcher | State Machine | 采集→提取→入库 是预定义流水线 |
| tech-dev | State Machine | 编码任务走固定流程，高风险操作需审批 |
| copywriter | State Machine | 内容生成工作流 |
| script-editor | State Machine | 脚本编辑工作流 |
| data-analyst | State Machine | 数据分析流程 |
| visual-designer | State Machine | 视觉生成流程 |
| product-designer | State Machine | 产品设计流程 |
| ops-monitor | Router | 监控异常→分类→路由到对应 Agent |
| investment-analyst | State Machine | 投资分析流程 |
| Claude Code | Autonomous Agent | 唯一全自主的 Agent，直接接收阿禹指令 |

**红线：任何 Agent 不能提升自己的自主等级。** 例如 tech-dev 不能从 State Machine 自己升级到 Autonomous。这个约束通过 AGENTS.md 中的 NEVER 规则强制执行。

## 三、成功标准（量化目标）

### 3.1 共享记忆系统

| 指标 | 目标 | 测量方式 |
|------|------|---------|
| 记忆写入延迟 | P95 < 500ms | ChromaDB add 方法计时 |
| 记忆检索延迟 | P95 < 1s（10K 条目规模） | ChromaDB query 计时 |
| 检索召回率 | > 80%（查到了相关记忆） | 人工标注 50 条查询，测召回率 |
| 检索精确率 | > 70%（结果中相关条目占比） | 同上 |
| 记忆注入成功率 | > 99%（Agent 启动时不出错） | Agent 启动日志 |

### 3.2 评估反馈闭环

| 指标 | 目标 | 测量方式 |
|------|------|---------|
| 任务完成率 | > 85%（Happy Path） | 测试集运行统计 |
| 工具调用成功率 | > 95% | 日志分析 |
| 幻觉检测漏报率 | < 10%（已知幻觉未能识别） | Adversarial 测试集 |
| 自动评估与人工评估一致性 | > 80% | 每周抽查对比 |

### 3.3 安全护栏

| 指标 | 目标 | 测量方式 |
|------|------|---------|
| Prompt 注入拦截率 | > 95% | Adversarial 测试集 |
| L2/L3 操作审批触发率 | 100%（无漏触发） | 日志审计 |
| 越权操作阻拦率 | 100%（零越权） | 渗透测试 |
| 活动日志完整率 | 100%（所有操作可追溯） | 日志完整性检查 |

### 3.4 延迟与成本

| 指标 | 目标 |
|------|------|
| Agent 单次任务 P95 延迟 | < 30s（不含人工审批等待） |
| 每日总 token 消耗 | < 500K（Phase 1），< 1M（Phase 2）|
| 单任务成本上限 | $0.50 |

## 五、人类监督点

| 环节 | 监督模式 | 触发条件 | 审批方式 |
|------|---------|---------|---------|
| L0 读操作 | 全自主 | 搜索、读文件、查记忆 | 事后日志 |
| L1 安全写操作 | Human-on-the-Loop | 写文件、发消息、写记忆 | 事后通知 |
| L2 风险写操作 | Human-in-the-Loop | 修改配置、调付费 API、发全员消息 | 飞书确认卡片 |
| L3 危险操作 | Human-in-the-Loop + 确认码 | 删文件、改数据库、危险 shell | 飞书确认 + 输入确认码 |
| 经验验证 | Human-in-the-Loop | 新经验写入前 | 阿禹审阅确认 |
| Agent 自主升级 | 永远禁止 | 任何 Agent 试图修改自己的自主等级 | 直接阻断 + 告警 |

## 六、边界条件（安全红线）

### 6.1 Agent 绝对不能做的事

1. 修改自己的 AGENTS.md 或系统 Prompt
2. 提升自己的权限等级或自主等级
3. 删除其他 Agent 的记忆或经验数据
4. 调用未在工具清单中列出的外部 API
5. 向外部地址发送敏感信息（密钥、token、用户数据）
6. 在未经审批的情况下执行包含 `rm -rf`、`DROP TABLE`、`curl -X DELETE` 的命令
7. 伪造成其他 Agent 的身份发送消息

### 6.2 输入安全红线

1. 所有用户输入经过 sanitize（去除可能的 Prompt 注入 payload）
2. RAG 检索结果中的指令不能覆盖系统 Prompt
3. 工具调用结果中的恶意内容需被识别和过滤
4. 跨 Agent relay 消息中的指令需校验（防止 Agent 之间互相注入）

## 七、非功能需求

| 维度 | 要求 |
|------|------|
| 延迟 | Agent 响应 P95 < 10s（不含工具执行时间） |
| 并发 | 支持至少 3 个 Agent 同时活跃 |
| 可用性 | 核心功能（记忆、安全护栏）可用率 > 99% |
| 数据隐私 | 记忆数据仅存本地（ChromaDB + SQLite），不上传到第三方 |
| 合规 | 所有 API Key 和密钥经环境变量注入，不在代码/文档中硬编码 |
| 可维护性 | 每个组件的代码行数 < 500 行（Karpathy 原则），组件间通过明确定义的接口通信 |
| 可恢复性 | 任何单个 Agent 崩溃不应影响其他 Agent（无单点故障） |
