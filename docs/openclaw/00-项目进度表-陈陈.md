# AI Agent 完整系统搭建 — 项目进度表

> **编制人：** 陈陈（Central Agent）
> **编制日期：** 2026-06-04
> **项目范围：** 11个Agent × 6个核心组件的完整系统搭建
> **方法论来源：** Anthropic "Building Effective Agents"、LangChain State of Agent Engineering、OpenAI Agentic Governance

---

## 一、总体原则

本进度表**不是瀑布式甘特图**，而是以「评估-改进」循环为基本单元的迭代计划。每个阶段有明确的 Gate 退出标准，不通过则不进入下一阶段。

### 核心约束

| 约束项 | 说明 |
|--------|------|
| 评估先行 | 写任何 Agent 代码之前，先建立评估基准和测试数据集 |
| 弹性 buffer | 每阶段预留 30-40% 时间给 Prompt 调试和工具优化 |
| Gate 机制 | 每个工作流步骤之间设程序化检查点，不通过则阻断 |
| 观测基建 | 可观测性不是事后补课，是与业务代码同步交付的基建 |
| 轨迹评估 | 多步 Agent 必须评估整个执行路径，不能只看最终输出 |

> **目标：** 在写任何 Agent 代码之前，先把"怎么衡量好不好"这件事搞定。

### 里程碑

| # | 里程碑 | 可量化指标 | 负责人 | Gate 条件 |
|---|--------|-----------|--------|-----------|

### 评估框架设计

```
评估维度          指标                        工具/方法
──────────────────────────────────────────────────────────
Grounding       上下文引用准确率              LLM-as-Judge + 人工校准
                工具调用正确率                结构化 rubric

### 测试数据集分类（每组件）

| 类别 | 占比 | 说明 |
|------|------|------|
| Happy Path | 40% | 正常场景，预期输入输出明确 |

## 九、评估-改进循环节奏

每两周一次评估-改进循环（贯穿全项目）：

```
┌──────────────────────────────────────────────┐
│  评估-改进循环（每2周）                         │
│                                              │
│  1. 跑全量测试集（离线评估）                    │
│  2. 检查各项指标 vs 上期基线                    │

│  6. 更新基线值                                │
