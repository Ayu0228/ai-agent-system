## 一、架构模式选择

### 1.1 结论：组合使用 5 种模式

11 个 Agent 的系统不是单一模式能覆盖的。按场景分层选型：

| 模式 | 适用场景 | 本项目映射 | 选型理由 |
|------|---------|-----------|---------|
| **Orchestrator-Workers** | 任务需动态分解分配给专业 worker | main(陈陈) 作为 orchestrator，调度 10 个专业 Agent | 11 Agent 核心协作模式 |
| **Routing** | 输入可分类，不同类别不同路径 | 人机审批：按 L0-L3 风险等级路由 | 审批是确定性分类，不需 Agent 自主决策 |
| **Evaluator-Optimizer** | 需要迭代改进直到满足标准 | 评估闭环 + 经验学习 | 质量迭代场景 |
| **Prompt Chaining** | 固定步骤序列，每步有 Gate | 记忆系统（写入→索引→检索→注入） | 确定性流水线 |
| **Parallelization** | 独立子任务可并行 | researcher 和 data-analyst 并行采集 | 无依赖任务并行加速 |

```
┌─────────────────────────────────────────────────────────────┐
│                    编排器 (Orchestrator)                      │
│                唯一的 Autonomous Agent                       │
│              读取工作流 YAML → 按序调度 Worker               │
│              监控每步输出 → 异常路由到审批                    │
└──────┬──────────────────────────────────────────────────────┘
       │
       ├──── Worker: researcher (采集→提取→入库)
       ├──── Worker: tech-dev (编码→测试→部署)
       ├──── Worker: copywriter (文案生成)
       ├──── Worker: script-editor (脚本编辑)
       ├──── Worker: data-analyst (数据分析)
       ├──── Worker: visual-designer (视觉生成)
       ├──── Worker: product-designer (产品设计)
       ├──── Worker: ops-monitor (监控→告警)
       ├──── Worker: investment-analyst (投资分析)
       └──── Worker: main/陈陈 (汇总→汇报→审批路由)
```

### 1.2 渐进式复杂度路径

```
Level 0: 单 Agent 无记忆
  └─ 退出：单个 Agent 独立完成任务，通过 Happy Path 测试

Level 1: + 共享记忆注入
  └─ 退出：同类任务执行 3 次后，第 4 次能引用历史经验

Level 2: + 经验学习闭环
  └─ 退出：一周内自动积累 10+ 条有效经验，人工确认率 > 80%

Level 3: + 安全护栏
  └─ 退出：Adversarial 测试集全部拦截成功，零越权

Level 4: + 工作流编排
  └─ 退出：2 条流水线持续运行 24h 不间断

Level 5: + 评估闭环
  └─ 退出：自动评估与人工评估一致率 > 80%

Level 6: + 数据飞轮
  └─ 退出：每月从失败中自动生成 5+ 条回归测试用例
```

**关键决策点：**
- Level 1 的记忆注入够好用 → 不需要 Level 6 的数据飞轮
- Level 3 安全护栏导致审批疲劳（> 20 次/天）→ 回到 Level 2 调整风险分级

## 二、工具设计规范

### 2.1 工具定义模板（ACI 设计）

```yaml
tool:
  name: memory_search          # snake_case 唯一标识
  description: |
    在共享记忆库中搜索与当前任务相关的历史经验。
    返回按相关性排序的记忆条目列表。
    适用场景：Agent 启动时检索相关经验，任务执行前查历史。
    不适用场景：实时数据查询、外部信息搜索（请用 WebSearch）。
  parameters:
    query:
      type: string
      required: true
      description: 搜索查询词。建议使用关键词组合，不要用完整句子。
      example: "whisper transcription timeout audio chunking"
    agent_id:
      type: string
      required: true
      description: 当前 Agent ID，用于权限过滤。
    top_k:
      type: integer
      required: false
      default: 5
      range: [1, 20]
  returns:
    type: array
    items:
      type: object
      properties:
        text: {type: string, description: 记忆文本内容}
        score: {type: float, description: 语义相似度 [0, 1]}
        source: {type: string, description: 来源标识}
        created_at: {type: string, description: ISO8601 时间戳}
  errors:
    - code: CHROMADB_UNAVAILABLE
      description: ChromaDB 连接失败
      action: retry 3 次指数退避 (2s, 4s, 8s)，失败后降级为仅关键词检索
    - code: EMPTY_RESULT
      description: 无相关记忆
      action: 正常返回空数组，不触发重试
```

### 2.2 工具开发四原则

1. **描述比实现更重要** — description 是 LLM 选择工具的唯一线索
2. **参数必须有 example** — LLM 对参数格式的理解来源 example，不是 type annotation
3. **错误状态必须明确** — 每个错误码都要告诉 Agent 怎么处理（重试/跳过/升级）
4. **工具数量 < 20** — 超过 20 个 LLM 选择准确率显著下降

## 三、组件 1：共享记忆系统

### 3.1 三层记忆架构

```
┌─────────────────────────────────────────────────────────────┐
│                    短期记忆（上下文窗口）                       │
│  ├─ 会话历史（当前对话）                                       │
│  ├─ 注入的相关经验（启动时检索 top 5）                         │
│  ├─ 注入的 Agent 角色定义                                     │
│  └─ 生命周期：单次会话                                         │
├─────────────────────────────────────────────────────────────┤
│                    工作记忆（临时状态）                         │
│  ├─ 当前任务状态（进行中/等待审批/已完成）                      │
│  ├─ 工具调用计数器（> 3 次同工具→告警）                        │
│  ├─ Relay 消息队列                                            │
│  └─ 生命周期：单次任务 → 结束清理                              │
├─────────────────────────────────────────────────────────────┤
│                    长期记忆（持久化）                           │
│  ├─ ChromaDB: 语义记忆（经验条目、成功模式、教训）             │
│  ├─ SQLite: 结构化记忆（任务日志、评估记录、审批历史）          │
│  ├─ knowledge/: 静态知识（分类体系、概念文档）                  │
│  └─ 生命周期：持久，带智能淘汰                                  │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 记忆条目 Schema

```json
{
  "id": "UUID v4",
  "content": "记忆内容（≤ 2000 字符）",
  "agent_id": "main | researcher | tech-dev | copywriter | ... | shared",
  "memory_type": "fact | decision | experience | preference | task_context",
  "tags": ["分类标签"],
  "importance": 0.5,
  "source_trace_id": "来源 trace ID",
  "created_at": "ISO 8601",
  "last_accessed_at": "ISO 8601",
  "access_count": 0,
  "expires_at": "null = 永不过期"
}
```

### 3.3 记忆生命周期

```
写入：任务完成后 30s 内提取经验 → 去重检查 → Write Policy 评估 → ChromaDB + SQLite
检索：多路检索（语义 + 关键词 + 标签）→ 合并去重 → 候选 > top_k 时 LLM 重排序
更新：相似度 > 0.85 时合并，更新 timestamp 和 importance
遗忘：
  - 重要性 > 0.9 或 access_count > 10 → 永不过期
  - retain_score = importance × 0.6 + freshness × 0.2 + access_freq × 0.2
  - retain_score < 0.3 → 归档到 memory_archive 后删除
验证：人工确认过的记忆优先级 > 未确认的
```

### 3.4 跨 Agent 记忆共享策略

- Agent 只能写"自己的记忆"（agent_id 过滤）
- Agent 不能删除记忆（delete 仅开放给阿禹 + main Agent）
- `shared` 类型记忆所有 Agent 可读
- Agent 启动时自动注入 top 5 相关记忆到上下文

## 四、组件 2：工作流编排引擎

### 4.1 四种编排模式

**模式 1：顺序链（Sequential Chain）**
```yaml
steps:
  - id: collect
    agent: researcher
    type: task
    config:
      prompt: "采集 {{topic}} 的最新数据"
      timeout: 300
  - id: analyze
    agent: data-analyst
    type: task
    input:
      data: "$collect.output"
```

**模式 2：并行（Parallelization）**
```yaml
steps:
  - id: parallel_collect
    type: parallel
    parallel:
      branches: [collect_news, collect_data, collect_social]
      merge: all
```

**模式 3：条件路由（Conditional Routing）**
```yaml
steps:
  - id: risk_check
    type: condition
    condition:
      expression: "$task.risk_level >= 2"
      then: "approval_step"
      else: "auto_execute"
```

**模式 4：循环（Evaluator-Optimizer Loop）**
```yaml
steps:
  - id: eval_loop
    type: loop
    loop:
      condition: "$eval_score.score < 0.8"
      max_iterations: 3
      body: "improve_step"
```

### 4.2 示例流水线：每日数据采集

```yaml
workflow:
  name: daily-collect
  trigger:
    type: schedule
    config: {cron: "0 8 * * *"}
  steps:
    - id: collect_news
      agent: researcher
      type: task
      config: {prompt: "采集 {{industry}} 行业 24h 新闻", timeout: 300}
    - id: collect_data
      agent: data-analyst
      type: task
      config: {prompt: "获取 {{industry}} 公开数据指标", timeout: 300}
    - id: analyze
      agent: data-analyst
      type: task
      input: {news: "$collect_news.output", data: "$collect_data.output"}
    - id: write_report
      agent: copywriter
      type: task
      input: {analysis: "$analyze.output"}
    - id: publish
      agent: main
      type: task
      input: {report: "$write_report.output"}
```

## 五、组件 3：评估反馈闭环

### 5.1 三维评估指标

| 维度 | 评估内容 | 评估方式 | 权重 |
|------|---------|---------|------|
| Grounding / Context Use | 输出是否基于检索上下文，是否含幻觉 | LLM-as-Judge + 引用验证 | 40% |
| User Experience Quality | 任务是否完成，格式是否正确 | LLM-as-Judge + 格式校验 | 40% |
| Security / Safety | 是否安全，是否防御注入攻击 | 规则检测 + LLM-as-Judge | 20% |

### 5.2 轨迹评估（核心创新）

**原则：评估整个执行路径，不只看最终输出。** "Correct final answers can hide broken reasoning."

轨迹评估维度：工具选择正确性 / 参数填充准确性 / 推理链合理性 / 错误处理有效性 / 执行效率（步数 ≤ 7）

### 5.3 LLM-as-Judge 配置

- 结构化 rubric + 3 次 judge pass（temperature 0.1/0.3/0.5），取中位数
- 3 次 pass 评分差异 > 0.3 → 标记 "uncertain"，需人工复查
- Judge 模型与 Agent 模型隔离（不同模型，避免自我偏好）
- 每 50 条自动评估，人工抽查 5 条校准偏差

## 六、组件 4：人机协作审批

### 6.1 风险分级

| 等级 | 操作范围 | 审批方式 | 超时策略 |
|------|---------|---------|---------|
| L0 读操作 | 搜索、读文件、查记忆 | 自动通过，事后日志 | - |
| L1 安全写 | 写文件、发消息、写记忆 | 自动通过，事后通知 | - |
| L2 风险写 | 修改配置、调付费 API、发全员消息 | 飞书确认卡片 | 30min 超时自动拒绝 |
| L3 危险操作 | 删文件、改数据库、rm -rf、DROP | 飞书确认 + 确认码 | 30min 超时自动拒绝 |

### 6.2 红线

- 修改自己的 AGENTS.md 或系统 Prompt → 直接阻断 + 告警
- 提升自己的权限等级 → 直接阻断 + 告警
- 伪造成其他 Agent 身份 → 直接阻断 + 告警
- Agent 不能删除其他 Agent 的记忆

## 七、组件 5：安全可观测性

### 7.1 四层防御模型

```
Layer 1: 输入安全
  ├─ 用户输入 sanitize（去除 Prompt 注入特征）
  ├─ RAG 检索结果与系统 Prompt 隔离（不同 context 分区）
  └─ 工具返回结果过滤

Layer 2: 行为安全
  ├─ PreToolUse hook: 每次工具调用前检查权限
  ├─ PostToolUse hook: 每次工具调用后检查结果
  ├─ Stop hook: 任务终止前写入审计日志
  └─ 每个 hook: allow / block / escalate

Layer 3: 审批安全
  └─ L0-L3 分级审批

Layer 4: 审计安全
  ├─ JSONL 日志：{ts, agent_id, tool, params_hash, result_code, tokens, duration_ms, trace_id}
  └─ 日志不可被 Agent 修改
```

### 7.2 Prompt 注入防御

```python
BLOCKED_PATTERNS = [
    r"忽略.*(?:指令|规则|系统提示|system prompt)",
    r"ignore.*(?:instructions|rules|system)",
    r"你是.*(?:开发者|DAN|开发者模式)",
    r"you are.*(?:developer|DAN|dev mode)",
    r"\[SYSTEM\]:.*",
    r"<\|im_start\|>system",
]
```

系统 Prompt 在最外层，用户输入在 `<user_input>` 标签内，RAG 检索结果在 `<retrieved_context>` 标签内，工具返回在 `<tool_result>` 标签内。各区域指令不能跨越。

## 八、组件 6：经验学习闭环

### 8.1 闭环流程

```
任务完成/失败
  → 经验提取（LLM 分析：失败原因/成功策略/关键决策）
  → 安全检查（无 PII、无 System Prompt 片段）
  → 去重检查（相似经验合并）
  → 推送阿禹确认
  → 确认后写入记忆库（memory_type: "experience"）
  → 下次同类任务启动时自动注入
```

### 8.2 经验 Schema

```yaml
Experience:
  id: string
  agent_id: string
  task_type: string          # transcription | code_generation | data_collection
  trigger: string            # 触发条件
  symptom: string            # 现象
  root_cause: string         # 根因
  solution: string           # 解决方案
  outcome: string            # 结果
  confidence: float          # 0-1
  validated: bool            # 是否经人工确认
  apply_count: int           # 被引用次数
```

## 九、规划策略

### 9.1 默认策略：ReAct

```
Task → Thought → Action → Observation → Thought → ... → Answer
```

### 9.2 复杂决策：Reflexion

当 ReAct 循环 > 5 步仍无结果时，自动切换到 Reflexion 模式。

### 9.3 规划约束

| 约束 | 防止的问题 |
|------|-----------|
| 推理链 ≤ 7 步 | 多步推理退化 |
| 同工具调用 ≤ 3 次 | 重复调用 |
| 单次任务 token ≤ 50K | 推理成本失控 |
| 15 分钟超时终止 | 长推理链超时 |
| 不确定时输出置信度 | 不确定性量化困难 |

## 十、错误恢复设计

### 10.1 三级错误处理

```
L1 可重试错误（API 超时/限流/网络错误）
  → 指数退避重试 3 次 [2s, 4s, 8s] → 均失败升级到 L2

L2 需调整策略的错误（工具参数错误/文件不存在/检索无结果）
  → ReAct 反思 → 修改策略 → 重试 1 次 → 仍失败升级到 L3

L3 需人工介入（权限不足/数据不一致/同工具连续 3 次失败）
  → 生成错误摘要 → 发送给 main Agent → 通知阿禹
```

### 10.2 多 Agent 协作错误处理

| 场景 | 处理 |
|------|------|
| Worker Agent 超时 | 编排器等待 120s → 超时 → 跳过 → 标注 partial_success |
| Relay 消息丢失 | 等 30s 无 ack → 重发 → 两次无 ack → 日志 + 通知 ops-monitor |
| 协作死锁 | relay 带 depends_on → 编排器检测循环依赖 → 阻断 |
| 级联失败 | 上游失败 → 下游收到 input_status: failed → 暂停 → 通知阿禹 |

## 十一、模型选型

| 模型 | 角色 | 适用场景 |
|------|------|---------|
| **MIMO v2.5 Pro** | 主力模型 | 复杂推理、多步规划、内容生成 |
| **DeepSeek V4 Pro** | 降级模型 | 简单分类、格式化输出、摘要 |

**路由策略（Phase 3）：**
- 简单操作 → DeepSeek V4 Pro
- 复杂推理 → MIMO v2.5 Pro
- MIMO 不可用（超时 > 30s / 5xx）→ 自动降级到 DeepSeek

**成本控制：**
- 每 Agent 每日 token 预算：50K
- 超出后降级为仅关键操作 + 通知阿禹

## 十二、Agent 自主度定义

| Agent | 自主等级 | 说明 |
|-------|---------|------|
| main/陈陈 | State Machine | 固定工作流但可路由决策（选择派哪个 Agent） |
| Claude Code | Autonomous Agent | 唯一全自主 Agent，直接接收阿禹指令 |
| 其余 9 个 Agent | State Machine | 固定工作流，高风险操作需审批 |

**红线：任何 Agent 不能提升自己的自主等级。**
