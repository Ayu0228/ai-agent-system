## 零、渐进式复杂度路径

本项目采用 6 级渐进复杂度模型。每个 Phase 内部按此模型递进，每级有明确退出标准，验证不通过则不升级。

```
Level 0: 单 Agent 无记忆
  └─ 退出：单个 Agent 独立完成任务，通过 Happy Path 测试

Level 1: + 共享记忆注入
  └─ Agent 启动时注入 3 条相关历史经验
  └─ 退出：同类任务执行 3 次后，第 4 次能引用历史经验

Level 2: + 经验学习闭环
  └─ 任务完成/失败后自动提取经验 → 人工确认 → 写入记忆
  └─ 退出：一周内自动积累 10+ 条有效经验，人工确认率 > 80%

Level 3: + 安全护栏
  └─ 所有工具调用经 PreToolUse hook 拦截，L2/L3 操作触发飞书审批
  └─ 退出：Adversarial 测试集全部拦截成功，零越权

Level 4: + 工作流编排
  └─ YAML 定义流水线，编排器自动调度
  └─ 退出：2 条流水线持续运行 24h 不间断

Level 5: + 评估闭环
  └─ 每次任务后自动评分，阿禹抽查
  └─ 退出：自动评估与人工评估一致率 > 80%

Level 6: + 数据飞轮
  └─ 生产失败自动回流到标注队列
  └─ 退出：每月从失败中自动生成 5+ 条回归测试用例
```

---

## 一、总体原则

| 约束项 | 说明 |
|--------|------|
| 评估先行 | 写任何 Agent 代码之前，先建立评估基准和测试数据集 |
| 弹性 buffer | 每阶段预留 30-40% 时间给 Prompt 调试和工具优化 |
| Gate 机制 | 每个步骤间设程序化检查点，不通过则阻断 |
| 观测基建 | 可观测性与业务代码同步交付 |
| 轨迹评估 | 多步 Agent 评估整个执行路径，不只看最终输出 |

## 三、Phase 0：评估基建（第 1-2 周）

**目标：** 在写代码之前先建立"怎么衡量好不好"。
**对应 Level：** Level 0（单 Agent 无记忆）
**退出标准：** 评估框架就绪 + 测试数据集覆盖全部 6 组件。

| # | 里程碑 | 可量化指标 | Gate 条件 |
|---|--------|-----------|-----------|
| 0.1 | 评估框架选型与搭建 | 框架可运行，支持离线评估 + LLM-as-Judge | 能跑通一个 demo 评估用例 |
| 0.2 | 测试数据集 v1 | 每组件 ≥ 20 条用例（Happy Path 40% + Edge Cases 35% + Adversarial 25%） | 用例覆盖率评审通过 |
| 0.3 | 三维评估指标定义 | Grounding / UX Quality / Security 三维可计算 | 指标有基线值 |
| 0.4 | 质量基线记录 | 记录当前单次 LLM 调用的基准表现 | 基线数据完整可回溯 |

## 九、评估-改进循环

每两周一次评估-改进循环（贯穿全项目）：

```
1. 跑全量测试集（离线评估）
2. 检查各项指标 vs 上期基线
3. 识别退化项 → 定位根因
4. 制定改进计划（Prompt / 工具 / 架构调整）
5. 实施改进 → 下一轮评估验证
6. 更新基线值
```
