## 各文档采长补短方案

### 00-项目进度表

**Claude Code 版长处：**
- 6级渐进复杂度模型（Level 0→6），每级有退出标准
- 简洁明了，适合快速理解项目演进路径

**陈陈版长处：**
- 完整的22周4阶段计划，每阶段有详细里程碑和Gate条件
- 风险矩阵+Buffer规划（34%）
- 每2周评估-改进循环
- 具体的Agent组合验证计划

**合并策略：** 以陈陈版为骨架，补充Claude Code版的渐进复杂度模型作为"每个Phase内的演进逻辑"。

### 02-设计方案

**Claude Code 版长处：**
- 架构模式选择对比表（5种模式选型理由）
- 工具设计模板（含错误码和action定义）
- 规划策略约束（ReAct + Reflexion 切换条件）
- 4层安全防御模型

**product-designer 版长处：**
- 6个组件的完整设计，每个组件含：设计概述/为什么做/工作量估算/Schema定义/接口代码/实现逻辑
- 具体的YAML工作流示例（daily-collect / content-production）
- 双模型路由策略代码
- 记忆生命周期完整定义
- ChromaDB+SQLite建表DDL

**合并策略：** 以product-designer版为骨架，补充Claude Code版的架构对比表、工具设计模板规范、安全4层模型。

### 03-开发文档

**Claude Code 版长处：**
- 完整的目录结构
- 先API后框架的开发铁律
- 工具ACI定义模板
- 清晰的实现顺序（10步15天）
- 代码约定

**tech-dev 版长处：**
- 完整可运行的Python实现代码
- 上下文压缩器、记忆整合器的完整实现
- 问题库引用（每个组件标注了规避的问题编号）
- Hooks生命周期完整代码

**合并策略：** 以Claude Code版为结构，tech-dev版的实现代码作为附录参考。两者不冲突——Claude Code版是"规范"，tech-dev版是"参考实现"。

### 01-需求文档

**Claude Code 版长处：**
- 工具清单含完整 ACI Schema（输入/输出/错误码）
- 安全红线清晰（7条Agent绝对不能做的事 + 输入安全红线）
- 非功能需求量化明确（延迟P95<10s、并发3 Agent、可用率>99%）

**product-designer 版长处：**
- 每个组件的"为什么不用Agent"分析——解释为什么共享记忆/审批路由/安全观测用工作流而非Agent
- 11个Agent的详细角色定义（每个Agent的职责边界、输入输出、协作关系）
- 6大核心组件逐一需求分析（用户故事 + 验收标准）
- 技术栈明确（OpenClaw框架、ChromaDB、SQLite、MIMO v2.5 Pro）

**合并策略：** Claude Code版更侧重"工具接口定义"和"安全边界"，product-designer版更侧重"组件需求分析"和"Agent角色定义"。两者互补，Claude Code版已满足项目需求文档要求，product-designer版作为补充参考。

### 04-测试验收文档

**Claude Code 版长处：**
- 完整的评估哲学（轨迹评估 > 输出评估，评估先行原则）
- 三维评估指标定义清晰（Grounding 40% + UX 40% + Safety 20%）
- 三类测试用例YAML示例（HP/EC/AV各一条完整示例）
- 隐藏失败检测的5个自动标记条件
- 数据飞轮闭环流程和度量
- 阿禹验收Checklist

**tech-dev 版长处：**
- 6个组件逐一详细测试方案（每个组件含单元测试/集成测试/端到端测试）
- 自动化测试架构（pytest + CI集成）
- 已知问题库映射（每个测试项标注了规避的问题编号）
- 更细粒度的指标定义（如事实准确率≥95%、引用溯源率≥80%、幸运幻觉率≤5%）

**合并策略：** Claude Code版更侧重"评估框架设计方法论"，tech-dev版更侧重"逐组件测试实施方案"。Claude Code版已满足测试验收文档要求，tech-dev版可作为测试实施时的详细参考。
